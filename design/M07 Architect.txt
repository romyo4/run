# M07 Architect

## 1. Overview

Architect は、AI Development Pipeline において、Planner が作成した Execution Plan を基に、実装可能なシステム設計を作成するモジュールである。

本モジュールの目的は、要求を実装へ直接変換するのではなく、実装可能な設計書へ変換することである。

Architect は**設計のみ**を担当し、コード生成・レビュー・Pull Request 作成は行わない。

### 適用対象

- 新規機能設計
- 機能追加設計
- リファクタリング設計
- バグ修正設計
- モジュール設計
- API設計

### 対象外

- 要求分析
- コード生成
- テスト実装
- Pull Request作成
- コードレビュー
- GitHub操作

---

# 2. Responsibilities

## 2.1 担当

Architect は以下を担当する。

- Execution Plan の分析
- システム設計
- モジュール設計
- API設計
- データ構造設計
- 実装方針決定
- Design Document 作成

---

## 2.2 担当しない

以下は本モジュールの責務外とする。

- 要求分析
- Task分解
- コード生成
- Pull Request作成
- コードレビュー
- Workflow実行

---

# 3. Design

## 3.1 入力

Architect が受け付ける入力。

```text
workflow_id
execution_plan
knowledge
project_context
architecture_guidelines
```

---

## 3.2 設計分析

Architect は以下を分析する。

- 要件
- 既存アーキテクチャ
- 既存モジュール
- 再利用可能コンポーネント
- 技術的制約

---

## 3.3 設計方針

設計では以下を優先する。

- Business First
- MVP First
- Single Responsibility
- Reuse First
- Simplicity

既存設計を変更する場合は理由を明記する。

---

## 3.4 Design Document

Architect が生成する成果物。

```text
Design ID

Objective

Architecture

Module Design

Interface Design

Data Structure

Implementation Strategy

Constraints
```

---

# 4. Constraints

## 4.1 Architectは実装しない

Architect は

- コード生成
- GitHub操作
- Pull Request作成

を行ってはならない。

---

## 4.2 Architectは要求を変更しない

Planner が作成した要求を変更してはならない。

要求変更が必要な場合は Planner に差し戻す。

---

## 4.3 Architectはレビューしない

設計の品質評価は Reviewer が担当する。

Architect 自身はレビュー結果を決定しない。

---

## 4.4 再利用優先

新規モジュールを追加する前に、

- 既存モジュール
- 既存インターフェース
- 既存設計

の再利用を検討する。

---

## 4.5 Logging

以下を記録する。

```text
timestamp

workflow_id

design_id

module_count

interface_count

result
```

Secretは出力しない。

---

# 5. Design Audit

## 5.1 責務確認

- 設計のみ
- Design Document作成のみ

要求分析・実装・レビューを含まない。

**判定：OK**

---

## 5.2 Foundation整合性

**F00**

- Design Before Code
- Single Responsibility
- Reuse First
- Simplicity

に適合。

**判定：OK**

---

**F01**

Design Domain を生成する。

**判定：OK**

---

**F02**

Common Interface を利用する。

**判定：OK**

---

**F03**

Configuration を参照可能。

**判定：OK**

---

## 5.3 重厚壮大化監査

以下はMVP対象外。

- 自動アーキテクチャ最適化
- マイクロサービス分割AI
- パフォーマンスシミュレーション
- コスト最適化AI
- UML自動生成
- Infrastructure設計
- Database自動最適化

**判定：削除済み**

---

## 5.4 実装可能性

Architect は Execution Plan を Design Document に変換する単一責務のモジュールとして実装可能である。

設計成果物のみを生成し、後続の Executor がその成果物を入力として利用する。

**判定：実装可能**

---

# 6. Summary

Architect は AI Development Pipeline の設計モジュールである。

責務は**「Execution Plan を Design Document に変換すること」**に限定し、コード生成・レビュー・GitHub操作には関与しない。

MVPでは設計品質と責務分離を重視し、Design Before Code の原則を実現する。
