# M04 Permission Manager

## 1. Overview

Permission Manager は、AI Development Pipeline における操作権限を管理するモジュールである。

本モジュールの目的は、各モジュールが実行しようとする操作について、実行可否を判定することである。

本モジュールは**権限の判定のみ**を担当し、認証や業務処理は行わない。

### 適用対象

- Planner
- Designer
- Executor
- Reviewer
- Scheduler
- Command Router

### 対象外

- ユーザー認証
- OAuth処理
- ログイン管理
- Secret管理
- 業務ロジック

---

## 2. Responsibilities

### 2.1 担当

Permission Manager は以下を担当する。

- 操作権限の判定
- モジュールごとの許可操作管理
- 判定結果の返却

### 2.2 担当しない

以下は本モジュールの責務外とする。

- Task生成
- Workflow管理
- Configuration管理
- Knowledge管理
- GitHub操作
- Slack送信
- Discord送信

---

## 3. Design

### 3.1 管理対象

Permission Manager が管理する対象は以下とする。

- Module
- Operation

ユーザー情報や認証情報は保持しない。

---

### 3.2 Permission Model

Permission は以下の組み合わせで管理する。

```text
Module
Operation
Allow / Deny
```

---

### 3.3 公開インターフェース

#### check_permission()

**概要**

指定された Module が Operation を実行可能か判定する。

**入力**

```text
module
operation
```

**出力**

```text
allowed : boolean
reason : string
```

---

#### list_permissions()

**概要**

指定 Module に許可されている Operation 一覧を取得する。

**入力**

```text
module
```

**出力**

```text
Operation List
```

---

#### reload()

**概要**

Permission定義を再読込する。

MVPではシステム起動時のみ利用する。

---

### 3.4 権限一覧

| Module | 許可される Operation |
|---------|----------------------|
| Planner | ExecutionPlan作成 |
| Designer | Design作成 |
| Executor | Pull Request作成 |
| Reviewer | Review作成 |
| Scheduler | Workflow開始 |
| Knowledge Manager | Knowledge更新 |
| Command Router | Command振り分け |

---

### 3.5 判定フロー

```text
Module
    │
    ▼
Operation
    │
    ▼
Permission Manager
    │
    ▼
Allow / Deny
```

Permission Manager は判定のみを行い、実際の処理は呼び出し元モジュールが実行する。

---

## 4. Constraints

### 4.1 権限判定専用

Permission Manager は以下を実行してはならない。

- GitHub API呼び出し
- Slack送信
- Discord送信
- Codex実行
- Workflow変更
- Task変更

---

### 4.2 判定方式

MVPでは

**Module × Operation**

のみで判定する。

以下は採用しない。

- Role
- Group
- User
- Inheritance
- Policy Engine

---

### 4.3 フェイルセーフ

Permission情報が取得できない場合は

```text
Deny
```

を返す。

許可側へ倒してはならない。

---

### 4.4 ログ

判定結果はログへ出力する。

出力項目

```text
timestamp
module
operation
result
reason
```

Secret情報は出力しない。

---

## 5. Design Audit

### 5.1 責務確認

- 権限判定のみに責務を限定している
- 認証処理を含まない
- 業務ロジックを含まない

**判定：OK**

---

### 5.2 Foundation整合性

**F00**

- Simplicity
- Safety
- Single Responsibility

と整合する。

**判定：OK**

---

**F01**

Domain Model を変更しない。

**判定：OK**

---

**F02**

共通Interfaceから利用可能。

**判定：OK**

---

**F03**

Permission定義はConfiguration Manager経由で取得できる。

**判定：OK**

---

### 5.3 重厚壮大化監査

以下はMVP対象外とする。

- RBAC
- ABAC
- Policy Engine
- LDAP
- OAuth
- SSO
- Organization管理
- 動的ポリシー評価
- Attribute Based Permission

**判定：削除済み**

---

### 5.4 実装可能性

Module × Operation の判定テーブルのみで実装可能。

外部ライブラリへの依存は不要。

MVPとして十分に小さく、将来RBAC等へ拡張できる設計となっている。

**判定：実装可能**

---

## 6. Summary

Permission Manager は、AI Development Pipeline 全体の権限制御を担当するモジュールである。

本モジュールは**「判定のみ」を責務**とし、認証・業務処理・外部サービス操作は担当しない。

MVPでは **Module × Operation** の単純な判定方式を採用することで、実装容易性・保守性・拡張性を確保する。
