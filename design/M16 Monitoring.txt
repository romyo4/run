# M16 Monitoring

## 1. Overview

Monitoring は、AI Development Pipeline において、各モジュールおよび Workflow の稼働状況を継続的に監視し、システムの健全性を把握するモジュールである。

本モジュールの目的は、障害・異常・性能低下・失敗を早期に検知し、安定した運用を支援することである。

Monitoring は**監視および状態収集のみ**を担当し、要件分析・設計・実装・レビュー・通知内容の判断は行わない。

---

## 適用対象

- Workflow監視
- Module監視
- Job監視
- Health Check
- Performance監視
- Error監視

---

## 対象外

- 要件分析
- システム設計
- コード生成
- テスト
- Pull Request作成
- レビュー
- Workflow実行

---

# 2. Responsibilities

## 2.1 担当

Monitoring は以下を担当する。

- Health Check
- Module状態監視
- Workflow状態監視
- Error収集
- Performance収集
- Metrics収集
- Monitoring Report生成

---

## 2.2 担当しない

以下は本モジュールの責務外とする。

- Workflow制御
- コード修正
- Pull Request作成
- 通知文生成
- レビュー
- Business判断

---

# 4. Constraints

## 4.1 Monitoringは修正しない

Monitoring は

- Workflow修正
- コード修正
- Design変更

を行ってはならない。

---

## 4.2 Monitoringは通知しない

Monitoring は

通知内容

通知チャネル

を決定しない。

通知は Notification モジュールが担当する。

---

## 4.3 Read Only

Monitoring は

システム状態を変更してはならない。

取得のみ行う。

---

## 4.4 閾値

閾値は Configuration により管理する。

例

```text
Execution Time > 10 min

Failure Rate > 20%

Retry Count > 3
```

---

## 4.5 Logging

以下を記録する。

```text
timestamp

workflow_id

module

health_status

execution_time

failure_rate

warning_count
```

Secret・Access Token・Credential はログへ出力してはならない。

## 5.4 実装可能性

Monitoring は Workflow・各モジュール・システムメトリクスを収集し、Health Status と Monitoring Report を生成する単一責務のモジュールとして実装可能である。

異常検知後の通知は Notification モジュールへ委譲することで責務を分離する。

**判定：実装可能**

# 6. Summary

Monitoring は AI Development Pipeline の監視モジュールである。

責務は**「システムと Workflow の状態を継続的に監視し、健全性を可視化すること」**に限定する。

実装・レビュー・通知とは責務を分離し、システム全体の稼働状況を一元的に監視する。

代表例：

- Executor が10分以上応答しないことを検知する
- Tester の失敗率が閾値を超えたことを検知する
- Weekly Reviewer が予定時刻に実行されなかったことを検知する
- Workflow 全体の成功率・平均実行時間・失敗件数を集計する

これにより、開発パイプラインを安定して運用し、障害や性能低下を早期に発見できる。
