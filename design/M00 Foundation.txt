# M00 Foundation

## 1. Overview

Foundation は、AI Development Pipeline の全モジュールが共通で利用する基盤ライブラリである。

本モジュールの目的は、State/Domain Model・共通Interface・エラー体系・ログ規約・設定取得パターンなど、他モジュールが個別に再定義すべきでない共通要素を一箇所に集約し、全モジュールの整合性(単一責務・循環参照なし・重複実装なし)を土台から保証することである。

Foundation は**共通定義の提供のみ**を担当し、業務ロジック・Workflow制御・外部サービス連携は行わない。

他モジュールの「Design Audit」章にある **F00〜F03** は、すべて本モジュールが定義する下記4区分を指す。

- **F00** : Foundation Principles(設計原則)
- **F01** : Domain Model(共通データモデル)
- **F02** : Common Interface(共通インターフェース)
- **F03** : Configuration Access Pattern(設定取得パターン)

### 適用対象

- 共通データモデル定義
- 共通インターフェース定義
- 共通エラー・例外体系
- 共通ログ規約
- 共通バリデーション
- バージョン管理定数

### 対象外

- 業務ロジック
- Workflow制御
- 外部API呼び出し
- Configuration実体管理(M17が担当)
- Knowledge実体管理(M03が担当)

---

# 2. Responsibilities

## 2.1 担当

Foundation は以下を担当する。

- F00: 設計原則カタログの提供
- F01: 共通Domain Model(dataclass)の定義
- F02: 共通Interface(抽象基底クラス)の定義
- F03: Configuration取得パターン(呼び出し規約)の定義
- 共通Result型・エラー型・例外型の提供
- 共通ログ初期化ユーティリティの提供
- 共通バリデーションユーティリティの提供
- バージョン定数の管理

---

## 2.2 担当しない

以下は本モジュールの責務外とする。

- Task状態の実管理(State Managerが担当)
- Configuration値の実管理(Configuration Managerが担当)
- Knowledge本文の実管理(Knowledge Managerが担当)
- Workflow実行制御
- 外部サービスとの通信

---

---

## 3.2 F00: Foundation Principles

全モジュールの「Design Audit」F00判定は、以下の原則カタログから該当するものを引用する。モジュールは自身に該当する原則のみを選択して記載し、原則自体はFoundationが唯一の定義元とする。

|原則|意味|
|---|---|
|Single Responsibility|モジュールは単一責務のみを持つ|
|Design Before Code|設計完了前に実装を開始しない|
|MVP First|必要最小限の構成を優先する|
|Simplicity|複雑な仕組みを持ち込まない|
|Automation First|人手を介さず自動実行できる設計を優先する|
|Business First|事業目的に沿った判断を優先する|
|Quality First|品質基準を満たさない成果物を後工程へ渡さない|
|Documentation First|成果物は文書化された形式で引き渡す|
|Safety|失敗時は安全側(Deny/Fail)に倒す|
|Traceability|すべての操作をログとして追跡可能にする|
|Adapter Pattern|外部サービス差異を吸収する層を経由する|

新しい原則を追加する場合は、Design Freeze前の設計監査時のみ本表に追加できる。Design Freeze後の追加は禁止する。

---

## 3.3 F01: Domain Model

Foundation は `types.py` にて、全モジュール共通の Domain Model を dataclass として定義する。

|Domain|概要|主な利用モジュール|
|---|---|---|
|Task|要求単位のタスク|State Manager, Task Queue, Planner|
|SubTask|Taskの分解単位|Task Queue, Planner|
|Workflow|Task群の実行単位|State Manager, Scheduler, Command Router|
|Design|設計成果物|Architect, Design Auditor|
|Implementation|実装成果物|Executor|
|TestResult|テスト結果|Tester|
|PullRequest|Pull Request成果物|PR Creator, Executor, GitHub Manager|
|Review|レビュー結果|Reviewer, Weekly Reviewer|
|Knowledge|知識文書|Knowledge Manager|
|Context|AI実行時コンテキスト|Context Manager|
|Configuration|設定値|Configuration Manager|
|Notification|通知内容|Notification|
|CommunicationMessage|外部連携メッセージ|Slack/Discord Connector|
|Repository|Repository/Branch/Commit/File/Diff情報|GitHub Manager, Executor, PR Creator|

各Domainは以下の共通属性を持つ。

```text
id
created_at
updated_at
metadata
```

モジュール固有の属性はモジュール側の詳細設計書で追加定義する。Foundationは共通属性の型・命名規約のみを保証する。

---

## 3.4 F02: Common Interface

Foundation は `base_module.py` / `interfaces.py` にて、全モジュールが実装すべき共通インターフェースを定義する。

### BaseModule(抽象基底クラス)

```text
class BaseModule(ABC):
    def name(self) -> str
    def health_check(self) -> Result[bool]
```

全モジュールは `BaseModule` を継承し、自身の責務範囲のメソッドのみを追加する。

### Result型

```text
Result[T]
---------
success: bool
value: T | None
error: FoundationError | None
```

モジュール間のAPI戻り値は原則として `Result[T]` でラップする。真偽値のみを返す判定系API(例: Permission Managerの`check_permission()`)は、`Result[bool]` として扱う。

---

## 3.5 F03: Configuration Access Pattern

Foundation は Configuration の**取得方法(呼び出し規約)のみ**を定義し、設定値そのものは管理しない。設定値の実体管理は Configuration Manager(M17)を唯一の参照元とする。

```text
config = configuration_client.get(module_name: str, key: str) -> Result[Any]
```

- Foundationは `ConfigurationClient` インターフェースのみを提供する(インスタンスメソッドとして実装する。統合レビューにより、初版の`@staticmethod`宣言は大多数のモジュール実装・Configuration Manager実体と整合しないことが判明し是正した。CHANGELOG.md参照)
- 実装はConfiguration Managerが提供する
- 各モジュールはConfigurationClientの**インスタンス**をDIで受け取り、`config_client.get(...)`の形で呼び出す
- Foundation自身は設定値をキャッシュ・保持しない

---

## 3.6 エラー・例外体系

`errors.py` / `exceptions.py` にて以下を定義する。

```text
FoundationError (基底)
├── ValidationError
├── NotFoundError
├── PermissionDeniedError
├── StateTransitionError
├── ConfigurationError
└── ExternalServiceError
```

各モジュールは必要に応じてこれらを継承し、モジュール固有の例外を定義できる。新しい基底例外の追加はFoundation側でのみ行う。

---

## 3.7 ログ規約

`logger.py` にて、全モジュール共通のログ初期化関数を提供する。

```text
get_logger(module_name: str) -> Logger
```

- 標準ライブラリ `logging` のみを使用する(コーディング規約と整合)
- 出力形式: `timestamp | module_name | level | message`
- Secret・Token・Credential・Knowledge本文など機密情報はログへ出力しない(各モジュール側の責務)

---

## 3.8 バリデーション

`validation.py` にて、共通バリデーション関数を提供する。

```text
require_not_none(value, field_name)
require_non_empty(value, field_name)
require_in(value, allowed_values, field_name)
```

各モジュールはこれらを利用して入力検証を行い、独自のバリデーション基盤を重複実装しない。

---

## 3.9 バージョン管理

`version.py` にて、Design Freezeバージョンを管理する。

```text
DESIGN_VERSION = "v1.0"
```

Design Freeze後の仕様変更は本定数の更新(v1.1, v1.2, …)によってのみ表現する。

---

module_name

## 3.11 処理フロー

```text
全モジュール
     │
     ▼
Foundation (F00-F03)
     │
     ├── Domain Model提供 ──► 各モジュールのデータ構造
     ├── Common Interface提供 ──► 各モジュールの基底クラス
     ├── Error/Exception提供 ──► 各モジュールの例外処理
     └── Logger/Validation提供 ──► 各モジュールの横断的関心事
```

Foundationはいずれの方向にも業務データを送信しない。参照される専用モジュールである。

# 4. Constraints

## 4.1 Foundationは業務処理をしない

Foundation は

- Task状態管理
- Configuration実体管理
- Knowledge実体管理
- 外部API呼び出し

を行ってはならない。

---

## 4.2 Foundationは他モジュールに依存しない

Foundation は全モジュールの依存の起点(依存グラフの最下層)であり、他のいかなるモジュールにも依存してはならない。

---

## 4.3 Domain Model変更の制約

Design Freeze後、`types.py` の既存Domain定義(属性の削除・型変更)を行ってはならない。属性追加が必要な場合は次バージョンで対応する。

---

## 4.4 Logging

Foundation自体の内部動作(Logger初期化失敗等)のみを記録する。

```text
timestamp
module_name
event
result
```

---

# 5. Design Audit

## 5.1 責務確認

- 共通定義の提供のみ
- 業務ロジックを含まない
- 他モジュールへの依存を持たない

**判定：OK**

---

## 5.2 Foundation整合性

Foundation自身がF00〜F03の定義元であるため、本項は「自己整合性」として扱う。

- F00: 原則カタログを本モジュールが管理する
- F01: Domain Modelを本モジュールが管理する
- F02: Common Interfaceを本モジュールが管理する
- F03: Configuration Access Patternを本モジュールが管理する

**判定：OK**

---

## 5.3 重厚壮大化監査

以下はMVP対象外。

- プラグインアーキテクチャ
- 動的Domain Model生成
- スキーマレジストリ
- 分散トレーシング基盤
- イベントソーシング基盤
- 汎用DIコンテナ

**判定：削除済み**

---

## 5.4 実装可能性

Foundation は dataclass・ABC・標準`logging`ライブラリのみで実装可能であり、外部ライブラリへの依存を必要としない。

全モジュールが参照する土台として最初に実装対象とすべきモジュールである。

**判定：実装可能**

---

# 6. Summary

Foundation は AI Development Pipeline 全モジュールの共通基盤モジュールである。

責務は**「F00(原則)・F01(Domain Model)・F02(Common Interface)・F03(Configuration Access Pattern)を定義し、全モジュールへ提供すること」**に限定する。

Foundationは他のいかなるモジュールにも依存せず、依存グラフの最下層に位置することで、循環参照を構造的に防止する。

代表例：

- State Managerは`Task` Domain(F01)を利用してタスク状態を管理する
- Permission Managerは`BaseModule`(F02)を継承して実装される
- Executorは`ConfigurationClient`(F03)経由でRepository・Codex・GitHub設定を取得する

これにより、全22モジュールが共通の型・インターフェース・エラー体系の上で一貫した実装を行えるようになる。
