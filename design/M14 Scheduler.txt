# M14 Scheduler

> **Design Freeze 注記**: Design Freeze監査により、Command Router(M05)との相互依存を解消し、依存方向は Scheduler→Command Router の一方向のみと確定した。SchedulerがCommand Routerから状態照会を受けることはない。経緯は`CHANGELOG.md`参照。

## 1. Overview

Scheduler は、AI Development Pipeline において、定期実行およびイベント実行される Workflow を管理・起動するモジュールである。

本モジュールの目的は、指定されたスケジュールまたはイベントに従って Workflow を開始することである。

Scheduler は**Workflow の起動管理のみ**を担当し、要件分析・設計・実装・レビュー・Pull Request 作成は行わない。

### 適用対象

- 定期実行（Cron）
- 手動実行
- イベント実行
- ワークフロー起動
- リトライ実行

### 対象外

- 要件分析
- システム設計
- コード生成
- テスト実行
- Pull Request作成
- GitHubマージ
- Workflow処理

---

# 2. Responsibilities

## 2.1 担当

Scheduler は以下を担当する。

- スケジュール管理
- イベント監視
- Workflow起動
- 実行キュー管理（MVPでは単一キュー）
- リトライ管理
- 実行履歴記録

---

## 2.2 担当しない

以下は本モジュールの責務外とする。

- Workflow処理
- コード生成
- Pull Request作成
- レビュー
- GitHub操作
- Business判断

---

# 4. Constraints

## 4.1 Schedulerは処理しない

Scheduler は

- Requirement Analysis
- Design
- Code Generation
- Test
- Review

を行ってはならない。

---

## 4.2 Workflowを変更しない

Scheduler は

Workflow の内容

Execution Plan

Design Document

を変更してはならない。

---

## 4.3 リトライ回数

MVPでは

```text
最大3回
```

まで自動リトライする。

超過した場合は失敗として記録する。

---

## 4.4 単一実行

MVPでは

同一 Workflow の重複起動を禁止する。

実行中の場合は新規起動しない。

---

## 4.5 Logging

以下を記録する。

```text
timestamp

workflow_id

trigger_type

execution_result

retry_count

duration
```

Secret・Access Token・Credential はログへ出力してはならない。

## 5.4 実装可能性

Scheduler は、手動・定期・イベントの各トリガーを受け取り、Workflow を起動する単一責務のモジュールとして実装可能である。

MVPでは単一キュー・単一実行・シンプルなリトライ機能に限定することで、実装容易性と保守性を確保する。

**判定：実装可能**

# 6. Summary

Scheduler は AI Development Pipeline の Workflow 起動管理モジュールである。

責務は**「Workflow を適切なタイミングで起動し、その実行状態を管理すること」**に限定する。

手動実行・定期実行・イベント実行を統一的に管理し、各 Workflow を Command Router へ引き渡すことで、自動化された開発パイプラインを実現する。

代表例：

- Slack から「LP改善を実装して」と指示されたら Workflow を起動する
- 毎週日曜日 20:00 に Weekly Reviewer (Fable) を自動実行する
- Pull Request のマージを検知して次の Workflow を起動する

これにより、日常の開発作業と定期的なプロジェクトレビューを自動的に実行できる。
