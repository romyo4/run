# M10 Tester

> **Design Freeze 注記**: Design Freeze監査によりExecutor(M09)との責務重複(テスト実行等)を是正し、Quality Gate PASS後のフローをPR Creator(M11)経由に修正した。経緯は`CHANGELOG.md`参照。

## 1. Overview

Tester は、AI Development Pipeline において、Executor が実装した成果物を検証し、設計どおりに実装されていること、および品質基準を満たしていることを確認するモジュールである。

本モジュールの目的は、Pull Request を Reviewer へ渡す前に、機械的に確認可能な品質を保証することである。

Tester は**テストの実行および結果の評価のみ**を担当し、実装・レビュー・マージ判定は行わない。

### 適用対象

- Unit Test
- Integration Test
- Regression Test
- Build Verification
- Lint Verification
- Static Analysis

### 対象外

- 要件分析
- システム設計
- コード生成
- テストコード生成
- Pull Request作成
- コードレビュー
- GitHubマージ

---

# 2. Responsibilities

## 2.1 担当

Tester は以下を担当する。

- テスト実行
- Build結果確認
- Lint結果確認
- Static Analysis実行
- Test Report作成
- 品質ゲート判定

---

## 2.2 担当しない

以下は本モジュールの責務外とする。

- 要件分析
- 設計変更
- コード修正
- Pull Request作成
- GitHub操作
- ビジネス判断
- レビューコメント作成

---

# 3. Design

## 3.1 入力

Tester が受け付ける入力。

```text
workflow_id

design_document

implementation_result

repository_information

test_configuration
```

---

## 3.2 実行項目

Tester は以下を実行する。

1. Build確認
2. Lint確認
3. Unit Test
4. Integration Test
5. Regression Test
6. Static Analysis
7. Test Report生成

---

## 3.3 成果物

Tester が生成する成果物。

```text
Build Report

Lint Report

Test Report

Static Analysis Report

Quality Gate Result
```

---

## 3.4 公開インターフェース

### execute_tests()

入力

```text
Implementation
```

出力

```text
Test Result
```

---

### validate_quality()

入力

```text
Test Result
```

出力

```text
Quality Gate Result
```

---

### publish_report()

入力

```text
Quality Gate Result
```

出力

```text
Test Report
```

---

## 3.5 品質ゲート

品質ゲートは以下を判定する。

|項目|条件|
|----|----|
|Build|Success|
|Lint|No Error|
|Unit Test|Pass|
|Integration Test|Pass|
|Regression Test|Pass|
|Static Analysis|Critical Errorなし|

すべて満たした場合のみ

```text
PASS
```

とする。

---

## 3.6 処理フロー

```text
Executor
      │
      ▼
Implementation
      │
      ▼
Build Check
      │
      ▼
Lint Check
      │
      ▼
Unit Test
      │
      ▼
Integration Test
      │
      ▼
Regression Test
      │
      ▼
Static Analysis
      │
      ▼
Quality Gate
      │
      ├── PASS ─────► PR Creator
      │
      └── FAIL ─────► Executor
```

> **Design Freeze 修正**: 従来「PASS ─────► Reviewer」としていたが、M11 PR Creatorの処理フロー定義(Tester→PR Creator→Reviewer)と矛盾していたため、PR Creatorを経由する形に修正した。経緯は`CHANGELOG.md`参照。

---

# 4. Constraints

## 4.1 Testerは修正しない

Tester は

- コード修正
- テストコード修正
- Design変更

を行ってはならない。

修正が必要な場合は Executor へ差し戻す。

---

## 4.2 Testerはレビューしない

Tester は

- コード品質評価
- ビジネス評価
- マージ判定

を行ってはならない。

---

## 4.3 自動実行

MVPではすべて自動実行する。

手動テストは対象外とする。

---

## 4.4 品質基準固定

品質ゲートは設定値に従う。

実行中に判定基準を変更してはならない。

---

## 4.5 Logging

以下を記録する。

```text
timestamp

workflow_id

build_result

lint_result

test_result

quality_gate

duration

result
```

Secret・Token・Credentialはログへ出力してはならない。

---

## 5.4 実装可能性

Tester は Executor が生成した成果物に対して Build・Lint・Test・Static Analysis を実行し、Quality Gate を判定する単一責務のモジュールとして実装可能である。

品質基準を満たした成果物のみを Reviewer へ引き渡す。

**判定：実装可能**

---

# 6. Summary

Tester は AI Development Pipeline の自動品質検証モジュールである。

責務は**「実装成果物を機械的に検証し、品質ゲートを通過したものだけを Reviewer へ渡すこと」**に限定する。

設計・実装・レビューとは責務を分離し、自動テストによる品質保証を担う。
