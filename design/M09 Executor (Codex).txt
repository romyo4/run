# M09 Executor (Codex)

> **Design Freeze 注記**: 本モジュールは Design Freeze 監査により、旧版で M10(Tester)/M11(PR Creator)と責務・Data Flowが重複していた点を是正した。テスト実行・Build実行・Lint実行・Pull Request作成の責務は削除し、Tester/PR Creatorへ一本化した。経緯は`CHANGELOG.md`参照。

## 1. Overview

Executor (Codex) は、AI Development Pipeline において、Architect が作成し、Design Auditor が承認した Design Document を基に実装を行うモジュールである。

本モジュールの目的は、承認済みの設計をコードおよびテストコードへ変換することである。テストの実行・品質判定・Pull Request作成は行わない。

Executor は**実装のみ**を担当し、要件分析・設計・設計監査・テスト実行・コードレビュー・Pull Request作成は行わない。

Codex は Executor の実装エンジンとして利用する。

### 適用対象

- 新規機能実装
- 機能追加
- バグ修正
- リファクタリング
- テストコード追加

### 対象外

- 要件分析
- システム設計
- 設計監査
- テスト実行(Testerが担当)
- Build実行(Testerが担当)
- Lint実行(Testerが担当)
- Pull Request作成(PR Creatorが担当)
- コードレビュー
- 優先順位決定
- GitHubマージ

---

# 2. Responsibilities

## 2.1 担当

Executor は以下を担当する。

- Design Documentの読込
- 実装コード生成
- ファイル編集
- テストコード生成
- 実装結果レポート作成

---

## 2.2 担当しない

以下は本モジュールの責務外とする。

- 要件分析
- Task分解
- 設計変更
- 設計監査
- テスト実行(Testerが担当)
- Build実行(Testerが担当)
- Lint実行(Testerが担当)
- Pull Request作成(PR Creatorが担当)
- PRレビュー
- GitHubマージ
- リリース判断

---

# 3. Design

## 3.1 入力

Executor が受け付ける入力。

```text
workflow_id

execution_plan

design_document

approved_design

project_context

repository_information
```

---

## 3.2 実装手順

Executor は以下の順序で処理する。

1. Design Document 読込
2. Repository取得
3. 実装対象特定
4. コード生成
5. ファイル更新
6. テストコード生成
7. 実装結果レポート作成

Build・Lint・Testの実行はTesterへ引き渡す。Pull Request作成はTesterの品質ゲートPASS後にPR Creatorが行う。

---

## 3.3 成果物

Executor が生成する成果物。

```text
Implementation

Modified Files

Generated Tests

Execution Report
```

---

## 3.4 公開インターフェース

### load_design()

入力

```text
Approved Design
```

出力

```text
Implementation Context
```

---

### implement()

入力

```text
Implementation Context
```

出力

```text
Implementation Result
```

---

## 3.5 処理フロー

```text
Architect
      │
      ▼
Design Auditor
      │
      ▼
Approved Design
      │
      ▼
Executor (Codex)
      │
      ▼
Implementation
      │
      ▼
Tester
      │
      ├── PASS ─────► PR Creator ─────► Reviewer
      │
      └── FAIL ─────► Executor
```

# 4. Constraints

## 4.1 Executorは設計しない

Executor は

- Architecture変更
- Module追加判断
- API設計変更
- Requirement変更

を行ってはならない。

設計変更が必要な場合は Architect へ差し戻す。

---

## 4.2 Executorはレビューしない

Executor は

- 品質判定
- マージ判定
- ビジネス判断

を行ってはならない。

---

## 4.3 承認済み設計のみ実装する

Design Auditor の承認を受けていない Design Document を実装してはならない。

---

## 4.4 Repository制約

Executor が変更できるのは対象 Repository のみとする。

複数 Repository を同時に変更してはならない(MVP)。

---

## 4.5 Executorはテスト実行・PR作成を行わない

Executor自身がBuild・Test・Lintを実行し、その結果に基づいてPull Requestを作成してはならない。

品質確認はTester、Pull Request作成はPR Creatorの責務とする(単一責務原則)。

---

## 4.6 Logging

以下を記録する。

```text
timestamp

workflow_id

design_id

modified_files

result
```

Secret・Token・Credential はログへ出力してはならない。

---

# 5. Design Audit

## 5.1 責務確認

- 実装のみ(コード生成・テストコード生成)

設計・テスト実行・レビュー・Pull Request作成を含まない。

**判定：OK**

---

## 5.2 Foundation整合性

**F00**

- Design Before Code
- Single Responsibility
- Automation First

に適合。

**判定：OK**

---

**F01**

`Implementation` Domain を生成する。

**判定：OK**

---

**F02**

Common Interface を利用する。

**判定：OK**

---

**F03**

Configuration を利用して Repository・Codex 設定を取得する。

**判定：OK**

---

## 5.3 重厚壮大化監査

以下はMVP対象外。

- 自動マージ
- 自動リリース
- マルチRepository実装
- 自動デプロイ
- 自動ロールバック
- Blue/Green Deployment
- Canary Deployment
- Infrastructure変更
- Database Migration管理
- Executor内でのテスト実行・PR作成の内製化

**判定：削除済み**

---

## 5.4 実装可能性

Executor は Design Document を入力として Codex を利用して実装コードとテストコードを生成する。

生成した実装はTesterへ引き渡し、Build・Test・Lintの実行と品質判定はTesterの責務とする。

**判定：実装可能**

---

# 6. Summary

Executor (Codex) は AI Development Pipeline の実装モジュールである。

責務は**「承認済み Design Document をコードおよびテストコードへ変換すること」**に限定する。

要件分析・設計・テスト実行・レビュー・Pull Request作成とは明確に責務を分離し、Design Auditor の承認を前提として実装を行うことで、**「Design Before Code」**と**単一責務**を確実に実現する。
